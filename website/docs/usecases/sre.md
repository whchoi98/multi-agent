# SRE편 (UC6-10)

## UC6. 프로덕션 P1 장애 대응 — `@mobilize`

### Pain Point

> *"금요일 오후 5시. 결제 서비스 API 응답시간이 30초를 넘겼습니다. PagerDuty가 울리고, Slack이 폭발하고, PM이 '언제 복구돼요?'라고 물어봅니다."*

P1 장애는 **MTTR(평균 복구 시간)이 곧 매출 손실**입니다.

### 3-Agent 분담

| Step | Agent | 질문 | 결과 |
|------|-------|------|------|
| 1 | **Kiro** | 어디를 봐야 하는가 | 5-Layer 트리아지 (인프라→네트워크→앱→외부→데이터) |
| 2 | **Codex** | 무엇이 잘못되었는가 | RDS 커넥션 풀 고갈 (450/450) ← commit def789의 N+1 쿼리 |
| 3 | **Gemini** | 어디까지 영향이 있는가 | 결제+주문 2개 서비스 영향, 3가지 복구 시나리오 |

### Master 판정

```
## 결론
Go — 옵션 A 즉시 실행 (롤백)

## 근거
- 3-Agent 모두 RDS 커넥션 고갈을 지목 → 높은 신뢰도
- 롤백 대상(4시간 전)이 충분히 최신이고 안정적
- 옵션 C(max_connections 증가)는 근본 해결 아님 → 기각

## 실행안
즉시 (MTTR 목표: 10분):
1. ECS 롤백: 이전 Task Definition revision 배포
2. RDS 커넥션 모니터링: 450 → 정상 수준(~150) 복귀 확인
3. P99 latency 모니터링: 30초 → 500ms 이하 복귀 확인

후속 (24시간 내):
4. N+1 쿼리 핫픽스 PR 생성 + 리뷰
5. 포스트모템 작성: N+1 탐지 CI 룰 추가 논의
```

:::tip Insight
`@mobilize`의 위력은 **병렬 사고**입니다. 사람 한 명이 Layer 1~5를 순서대로 점검하면 30분. 3-Agent가 각자 다른 레이어를 맡으면 10분.
:::

## UC7. SLO 기반 알림 설정 — `@verify`

### Pain Point

> *"알림이 하루에 200개 옵니다. 대부분 무의미한 CPU 60% 알림입니다. 진짜 중요한 알림은 노이즈에 묻힙니다."*

### Gemini vs Codex 비교

| 항목 | Gemini | Codex | 채택 |
|------|--------|-------|------|
| Error Budget 계산 | 월 43.2분 | 월 43.83분 | **Codex** (더 정밀) |
| Burn Rate 임계값 | 14.4x, 6x, 1x (3-tier) | 14.4x, 6x, 3x, 1x (4-tier) | **Codex** |
| 알림 채널 | Slack only | Slack + PagerDuty 분리 | **Codex** |
| 설명 가독성 | ★★★★★ | ★★★ | **Gemini** |

### 최종 알림 체계 (병합 버전)

| Tier | Burn Rate | Error Budget 소진 | 알림 | 채널 |
|------|-----------|-------------------|------|------|
| P1 | 14.4x | 1시간에 2% | 즉시 | PagerDuty |
| P2 | 6x | 6시간에 5% | 긴급 | Slack #incidents |
| P3 | 3x | 3일에 10% | 경고 | Slack #sre-alerts |
| P4 | 1x | 30일에 100% | 정보 | Slack #sre-metrics |

:::tip Insight
`@verify`에서 Gemini는 **"숲"**을, Codex는 **"나무"**를 봅니다. SLO 설정은 프레임워크(숲)와 수학적 정확성(나무) **둘 다 필요**하므로 교차검증이 최적입니다.
:::

| 순위 | 서비스 | 증가분 | 기여율 | 핵심 원인 |

## UC9. 재해복구(DR) 런북 작성 — `@design`

### Pain Point

> *"DR 훈련 날짜가 다음 주입니다. 런북이 2년 전에 작성된 것인데, 아키텍처가 완전히 바뀌었습니다."*

### Kiro의 설계 결과

**4-Phase DR 런북** — 현재 인프라의 실제 엔드포인트가 포함된 즉시 실행 가능 문서:

| Phase | 목표 시간 | 핵심 액션 |
|-------|----------|----------|
| 1. 감지 및 선언 | 5분 | Aurora lag 확인, DR 선언 |
| 2. DNS Failover | 10분 | Aurora Global DB failover, Route53 전환, ECS 스케일업 |
| 3. 검증 | 15분 | API Health, DB 쓰기, Redis 연결, E2E 테스트 |
| 4. 커뮤니케이션 | 즉시 | 상태 페이지, 고객 알림, 내부 보고 |

:::tip Insight
`@design` 모드에서 Kiro는 일반적 DR 런북이 아니라, **현재 인프라의 실제 엔드포인트와 명령어**가 포함된 즉시 실행 가능한 런북을 작성합니다.
:::

## UC10. 보안 감사 대응 자동화 — `@verify` → `@mobilize`

### Pain Point

> *"보안팀에서 SOC2 감사 체크리스트를 보내왔습니다. 87개 항목. 하나하나 수동으로 확인하면 일주일입니다."*

### 자동 승격

```
사용자 요청: @verify SOC2 감사 체크리스트 점검

Master 판단: 보안 감사 범위(87개) + 위험도 → @mobilize로 자동 승격
```

### 3-Agent 분담 결과

| 카테고리 | 항목 수 | Pass | Warning | Fail | 수동 |
|---------|---------|------|---------|------|------|
| IAM | 23 | 18 | 3 | **2** | 0 |
| 로깅 | 15 | 12 | 1 | 0 | 2 |
| 네트워크 | 18 | 15 | 2 | **1** | 0 |
| 암호화 | 12 | 10 | 0 | 0 | 2 |
| 컴플라이언스 | 19 | 8 | 0 | 0 | 11 |

**72개 자동 점검 완료**, 15개 수동 확인 필요, **3건 즉시 조치 필요** (미사용 IAM 키, Root MFA 미설정, 퍼블릭 SG)
