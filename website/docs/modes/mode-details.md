# 7가지 모드 상세

## 모드 전체 비교

| 모드 | 명칭 | 에이전트 | 실행 방식 | 적합한 작업 |
|------|------|---------|----------|-----------|
| `@ask` | 단독처리 | Claude | Master 단독 | 단순 질문, 파일 읽기/수정 |
| `@scan` | 속도우선 | Gemini | 단독 (45s) | 로그 분석, 비용 조회, 대량 데이터 |
| `@craft` | 정밀분석 | Codex | 단독 (90s) | 코드 변경, 테스트 작성, 설정 검증 |
| `@design` | 설계먼저 | Kiro | 단독 (120s) | 요구사항, 아키텍처, 런북 |
| `@verify` | 교차검증 | Gemini + Codex | 병렬 → 비교 | IAM, 보안, 코드 리뷰 |
| `@mobilize` | 총동원 | Kiro → Codex → Gemini | 순차 | 장애, 배포, DB 마이그레이션 |
| `@build` | 스펙→구현 | Kiro → Codex | 순차 | 리팩토링, API 마이그레이션 |

---

## 단독 모드

### @ask — 단독처리

**Master가 Slave 없이 직접 처리합니다.**

- 단순 질문, 파일 읽기/수정, 짧은 분석
- 자동 감지에서 어떤 키워드 패턴에도 매칭되지 않을 때 기본값
- Slave 오버헤드가 불필요한 작업

```
사용자: 이 파일에서 에러 원인이 뭐야?
→ [ask] Master가 직접 분석하여 답변
```

---

### @scan — 속도우선

**Gemini(Speed)가 대량 데이터를 빠르게 분석합니다.**

- CloudWatch 로그 10만 줄 패턴 분석
- AWS Cost Explorer 비용 이상 탐지
- Security Group 전수 검사
- 대량 리소스 목록 조회

```
사용자: @scan 최근 1시간 CloudWatch 로그에서 OOMKilled 패턴 분석해줘
→ Gemini가 37초 만에 108,432줄 분석 → 15분 주기 OOMKilled 패턴 발견
```

:::info scan 결과에서 보안 리스크가 감지되면
Master가 자동으로 `@scan → @verify`로 승격합니다. 빠른 스캔 결과에서 의심스러운 패턴이 발견되면 교차검증으로 확인합니다.
:::

---

### @craft — 정밀분석

**Codex(Precision)가 코드와 설정을 정밀하게 분석합니다.**

- 코드 변경, 리팩토링
- 유닛 테스트 생성 (비즈니스 로직 이해 기반)
- 오토스케일링 파라미터 튜닝
- 설정 파일 최적화

```
사용자: @craft ECS api-server 오토스케일링 설정 최적화해줘
→ Codex가 현재 설정 분석 → "70→60, cooldown 300→60" 구체적 숫자 제시
  + 비용 영향: 월 $180 → $240 (+$60)
```

---

### @design — 설계먼저

**Kiro(Spec)가 구현 전에 설계를 체계적으로 작성합니다.**

- 신규 마이크로서비스 경계 설계
- 재해복구(DR) 런북 작성
- 아키텍처 리팩토링 계획
- 요구사항 분석 및 태스크 분해

```
사용자: @design 알림 서비스를 마이크로서비스로 분리 설계해줘
→ Kiro가 의존성 분석 → 4-Phase 점진적 분리 전략 + 이벤트 스키마 설계
```

---

## 복합 모드

### @verify — 교차검증

**Gemini와 Codex가 병렬로 독립 분석 → Master가 비교 판정합니다.**

Master의 Judge 역할:
1. 두 Slave 결과의 **차이점** 식별
2. 각 차이점에 대해 어느 쪽이 더 적절한지 **근거와 함께** 판단
3. 필요시 두 결과를 **병합**하여 최적 버전 생성
4. 4-Block Format으로 최종 출력

```
사용자: @verify 이 IAM 정책을 검토해줘
→ Gemini: 3초 만에 kms:* 와일드카드 감지 (대안 제시 부족)
→ Codex: kms:* → kms:Decrypt, kms:GenerateDataKey 축소 권고 + Condition 블록 필요
→ Master: Codex가 더 정밀. 두 결과 병합하여 수정 정책 제시
```

| 적합한 작업 | 이유 |
|-----------|------|
| IAM 정책 변경 | 보안 이슈는 교차 확인 필수 |
| 코드 리뷰 | 넓게(Gemini) + 깊게(Codex) 상호 보완 |
| SLO 알림 설정 | 프레임워크(Gemini) + 수학 검증(Codex) |
| 보안 감사 | 패턴 매칭 + 로직 분석 조합 |

---

### @mobilize — 총동원

**3-Agent가 순차적으로 각자의 관점에서 분석 → Master가 Go/No-Go를 판정합니다.**

Master의 종합 역할:
1. 3단계(Kiro→Codex→Gemini) 결과를 **통합**
2. 단계 간 **모순이나 누락** 식별
3. **Go/No-Go 판정** + 통합 체크리스트 생성

```
사용자: @mobilize 결제 API P99 latency 30초 초과. 즉시 원인 분석 및 복구 방안
→ Step 1 Kiro: 5-Layer 트리아지 체크리스트
→ Step 2 Codex: 근본 원인 = commit def789의 N+1 쿼리
→ Step 3 Gemini: 결제+주문 2개 서비스 영향, 롤백이 최선
→ Master: Go — 즉시 롤백 실행 승인
```

| 적합한 작업 | 이유 |
|-----------|------|
| P1 장애 대응 | 분 단위 의사결정, 병렬 사고 |
| 프로덕션 DB 마이그레이션 | 데이터는 되돌릴 수 없다 |
| 보안 감사 대응 | 범위가 넓고 정확해야 함 |
| CI/CD 파이프라인 장애 | 다각도 원인 분석 |

---

### @build — 스펙→구현

**Kiro가 설계 → Codex가 구현 → Master가 갭을 분석합니다.**

Master의 검증 역할:
1. Kiro 스펙과 Codex 구현을 **대조**
2. 스펙 미충족 항목 식별
3. 구현 품질 판정 + 보완 사항 제시

```
사용자: @build VPC 모듈을 서브넷/NAT/라우팅 3개 모듈로 분리해줘
→ Step 1 Kiro: 변경 빈도 기반 모듈 분리 설계 + state 이동 계획
→ Step 2 Codex: 3개 모듈 코드 + migrate-state.sh 생성
→ Master: 충족률 95%, state backup 절차 보완 필요
```

| 적합한 작업 | 이유 |
|-----------|------|
| Terraform 리팩토링 | 설계 없는 리팩토링은 재앙 |
| 레거시 API 마이그레이션 | 스펙 없는 마이그레이션은 실패 |
| 신규 기능 구현 | 설계 → 구현 → 검증 파이프라인 |
