# 하이브리드 모드 시스템

> **평소엔 자동, 필요할 때만 수동.**

Multi-Agent CLI의 모드 시스템은 **하이브리드**입니다. 프롬프트만 입력하면 Master가 자동으로 최적 모드를 선택하고, 필요할 때 `@모드`로 강제 지정할 수 있습니다.

---

## 자동 감지 (기본 동작)

사용자가 모드를 지정하지 않으면, Master가 프롬프트의 **키워드 패턴**을 분석하여 최적 모드를 선택합니다.

### 키워드 매핑 (우선순위 순)

| 우선순위 | 키워드 패턴 | 선택 모드 | 이유 |
|---------|-----------|----------|------|
| 1 | 프로덕션 + (배포/롤백/삭제/마이그레이션/장애/P1/긴급) | `@mobilize` | 프로덕션 변경은 최대 신중 |
| 2 | (IAM/SG/보안/인증/암호화/키 관리) + (검토/변경/추가/정책) | `@verify` | 보안은 교차검증 필수 |
| 3 | (설계/아키텍처/요구사항/분리/런북/마이크로서비스) + 신규 | `@design` | 신규 설계는 Kiro 전문 |
| 4 | (리팩토링/마이그레이션/구현) + 설계 필요 판단 | `@build` | 설계→구현 파이프라인 |
| 5 | (로그/비용/대량/조회/요약) + 속도/긴급성 암시 | `@scan` | 대량 데이터는 속도 우선 |
| 6 | (코드/테스트/설정/수정/최적화) + 정밀/정확 필요 | `@craft` | 코드 정밀작업 |
| 7 | 위 패턴 미매칭 (기본값) | `@ask` | Master 단독 |

### 동작 예시

```
사용자: 이 IAM 정책 검토해줘

Master: [verify 모드] IAM 보안 변경 감지 → Gemini + Codex 교차검증으로 실행합니다.
        (다른 모드를 원하면 @모드를 붙여주세요)
```

```
사용자: 프로덕션 DB에 컬럼 추가해야 해

Master: [mobilize 모드] 프로덕션 + DB 변경 감지 → 4-Agent 총동원으로 실행합니다.
```

```
사용자: 이 함수 리팩토링해줘

Master: [craft 모드] 코드 수정 + 정밀 필요 → Codex 정밀분석으로 실행합니다.
```

## 수동 오버라이드 (`@모드`)

자동 감지를 무시하고 특정 모드를 **강제 지정**할 때 `@모드`를 프롬프트 앞에 붙입니다.

```
@scan 프로덕션 로그 빨리 확인해줘
  → 자동이면 mobilize이지만, scan 강제

@verify 이 코드 한 번 더 봐줘
  → 자동이면 craft이지만, verify 강제

@design 간단한 설정 변경인데 설계부터
  → 자동이면 ask이지만, design 강제
```

:::tip `@` 접두어를 사용하는 이유
`#`은 Markdown 헤딩과 충돌하므로, 모드 지정에는 `@`를 사용합니다.
:::

---

## 자동 감지 vs 수동 오버라이드

| 상황 | 권장 |
|------|------|
| 대부분의 작업 | **자동 감지** — Master가 알아서 판단 |
| 자동 감지가 과한 모드를 선택할 때 | **수동 오버라이드** — 더 가벼운 모드 강제 |
| 자동 감지가 가벼운 모드를 선택할 때 | **수동 오버라이드** — 더 신중한 모드 강제 |
| 데모/학습 목적 | **수동 오버라이드** — 특정 모드 동작 확인 |

:::info 자동 감지 후에도 Master가 모드를 조정할 수 있습니다
자동 감지 또는 수동 지정 후에도, 작업의 실제 위험도에 따라 Master가 [자동 승격/다운그레이드](/modes/escalation)를 적용합니다.
:::
