# 자동 승격/다운그레이드

자동 감지 또는 수동 지정 후에도, **작업의 실제 특성**에 따라 Master가 모드를 조정합니다.

---

## 승격 (더 신중하게)

위험도가 높은 작업이 감지되면 더 강력한 모드로 자동 승격합니다.

| 변경 | 트리거 조건 | 이유 |
|------|-----------|------|
| `@ask` → `@verify` | 보안 관련 코드 변경 (IAM, SG, 인증, 암호화, 키 관리) | 보안 변경은 교차검증 필수 |
| `@ask` → `@mobilize` | 프로덕션 배포/롤백, 데이터 삭제, 인프라 대규모 변경 | 프로덕션 변경은 최대 신중 |
| `@ask` → `@design` | 신규 기능 설계, 아키텍처 리팩토링, 마이크로서비스 분리 | 설계 없이 시작하면 재앙 |
| `@scan` → `@verify` | Gemini 결과에서 보안 리스크 감지 | 빠른 스캔에서 의심 발견 시 교차확인 |
| `@verify` → `@mobilize` | 교차 검증 중 심각한 불일치 발견 | 풀파이프라인으로 심층 분석 |

### 승격 시각화

```
                    ┌─── 보안 변경 ───→ @verify
                    │
    @ask ───────────┼─── 프로덕션 배포 ──→ @mobilize
                    │
                    └─── 신규 설계 ───→ @design

    @scan ─── 보안 리스크 감지 ──→ @verify

    @verify ─── 심각한 불일치 ──→ @mobilize
```

### 실제 예시

```
사용자: 이 설정 파일 수정해줘 (IAM 정책 포함)

Master: [ask 모드 → verify 모드 자동 승격]
        IAM 보안 변경 감지 → Gemini + Codex 교차검증으로 실행합니다.
```

## 다운그레이드 (더 효율적으로)

과도한 모드가 오히려 비효율적인 경우, 더 가벼운 모드로 조정합니다.

| 변경 | 트리거 조건 | 이유 |
|------|-----------|------|
| `@verify` → `@scan` | AWS 대량 읽기 조회 | Codex sandbox 네트워크 제한 |
| `@verify` → `@ask` | SSM 기반 트러블슈팅, 레거시 코드베이스 탐색 | Slave 타임아웃 문제, 파일 구조 파악 실패 |
| `@design` → `@ask` | 단순 설정 변경, 설계 불필요 | Kiro 오버헤드 |
| `@build` → `@craft` | 스펙 불필요한 단순 코드 변경 | 설계 단계 스킵 |
| `@mobilize` → `@verify` | 장애가 아닌 일반 배포, 위험도 낮은 변경 | 풀파이프라인 불필요 |

### 다운그레이드가 발생하는 주요 이유

:::info Codex sandbox 네트워크 제한
Codex는 sandbox 환경에서 동작하므로 외부 네트워크 접근이 제한됩니다. AWS API를 대량으로 호출해야 하는 작업(비용 분석, 리소스 조회 등)은 Codex 없이 Gemini 단독(`@scan`)이 더 효율적입니다.
:::

:::info Slave 타임아웃 문제
대규모 레거시 코드베이스를 탐색할 때, Slave가 타임아웃 내에 파일 구조를 파악하지 못하는 경우가 있습니다. 이때 Master가 직접 처리(`@ask`)하는 것이 더 빠릅니다.
:::

---

## Fallback

Slave 실행 중 문제가 발생했을 때의 안전장치입니다.

| 상황 | 동작 |
|------|------|
| Slave 타임아웃 | Master가 `@ask`로 fallback |
| Slave 2회 연속 실패 | 해당 Slave를 세션 동안 비활성화 |
| 전체 Slave 실패 | Master 단독 처리 + 사용자에게 상황 보고 |

```
Gemini (45초 초과)
    → Master: "Gemini 응답 시간 초과. 직접 분석으로 전환합니다."
    → @ask 모드로 동일 작업 수행
```

## 승격/다운그레이드 전체 흐름

```
사용자 프롬프트
    │
    ▼
자동 감지 (또는 @모드 수동 지정)
    │
    ▼
승격 규칙 적용
    ├─ 보안 관련? → verify 이상
    ├─ 프로덕션? → mobilize
    └─ 그 외 → 유지
    │
    ▼
다운그레이드 규칙 적용
    ├─ 네트워크 제한? → scan
    ├─ 단순 작업? → ask
    └─ 그 외 → 유지
    │
    ▼
최종 모드 결정 → 실행
```

:::tip 승격은 항상 다운그레이드보다 우선합니다
보안이나 프로덕션 관련 승격 조건이 동시에 다운그레이드 조건과 충돌하면, **승격이 우선**합니다. 안전은 효율보다 중요합니다.
:::
