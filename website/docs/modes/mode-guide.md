# 모드 선택 가이드

## 빠른 참조

```
"빨리 파악해야 해"          → @scan  (속도우선)
"정확해야 해"              → @craft (정밀분석)
"설계부터 해야 해"          → @design (설계먼저)
"두 번 확인해야 해"         → @verify (교차검증)
"절대 실패하면 안 돼"       → @mobilize (총동원)
"설계하고 바로 구현까지"     → @build (스펙→구현)
"간단한 거"               → @ask (단독처리)
```

---

## 상황별 의사결정 트리

```
작업이 들어왔다
    │
    ├─ 단순 질문/수정? ──── Yes ──→ @ask
    │
    ├─ 대량 데이터 분석? ── Yes ──→ @scan
    │
    ├─ 코드 정밀 작업? ─── Yes ──→ @craft
    │
    ├─ 신규 설계 필요? ─── Yes ──→ @design
    │
    ├─ 보안/리뷰 작업? ─── Yes ──→ @verify
    │
    ├─ 설계+구현 모두? ─── Yes ──→ @build
    │
    └─ 프로덕션/고위험? ── Yes ──→ @mobilize
```

---

## 역할별 자주 쓰는 모드

### DevOps 엔지니어

| 작업 | 권장 모드 | 이유 |
|------|----------|------|
| IAM 정책 변경 | `@verify` | 보안 이슈 교차검증 필수 |
| CloudWatch 로그 분석 | `@scan` | 대량 데이터 속도 우선 |
| Terraform 리팩토링 | `@build` | 설계 없는 리팩토링은 위험 |
| CI/CD 장애 대응 | `@mobilize` | 배포 차단 상황 |
| ECS 오토스케일링 튜닝 | `@craft` | 파라미터 정밀 분석 |

### SRE

| 작업 | 권장 모드 | 이유 |
|------|----------|------|
| P1 장애 대응 | `@mobilize` | 분 단위 의사결정 |
| SLO 알림 설정 | `@verify` | 수학적 정확성 + 프레임워크 |
| 비용 이상 탐지 | `@scan` | API 대량 호출 속도 |
| DR 런북 작성 | `@design` | 체계적 설계 필요 |
| 보안 감사 대응 | `@verify` → `@mobilize` | 자동 승격 가능 |

### 개발자

| 작업 | 권장 모드 | 이유 |
|------|----------|------|
| 레거시 API 마이그레이션 | `@build` | 스펙→구현 파이프라인 |
| 유닛 테스트 생성 | `@craft` | 비즈니스 로직 이해 기반 |
| PR 코드 리뷰 | `@verify` | 넓게 + 깊게 상호 보완 |
| 마이크로서비스 설계 | `@design` | 경계 결정이 핵심 |
| DB 마이그레이션 | `@mobilize` | 데이터는 되돌릴 수 없다 |

## 모드 선택 안티패턴

| 안티패턴 | 문제 | 대안 |
|---------|------|------|
| 단순 질문에 `@mobilize` | 3-Agent 오버헤드만 발생 | `@ask` 사용 |
| 보안 리뷰에 `@scan` | 속도만 빠르고 깊이 부족 | `@verify` 사용 |
| 프로덕션 변경에 `@craft` | 단일 관점, 영향도 분석 없음 | `@mobilize` 사용 |
| 코드 수정에 `@design` | Kiro는 설계 전문, 코드 변경은 역할 외 | `@craft` 사용 |

:::tip 모드를 잘못 선택해도 괜찮습니다
Master의 [자동 승격/다운그레이드](/modes/escalation) 규칙이 위험한 선택을 보정합니다. 보안 작업에 `@ask`를 선택하면 Master가 자동으로 `@verify`로 승격합니다.
:::
