# 에이전트 구성

Multi-Agent CLI는 **1 Master + 3 Slave** 구조입니다. 각 에이전트는 고유한 강점을 가지며, Master가 작업 특성에 따라 최적의 조합을 선택합니다.

---

## Claude — Master / Judge

| 항목 | 내용 |
|------|------|
| **역할** | 오케스트레이션, 최종 판정, 복합 추론 |
| **강점** | 대규모 컨텍스트 처리, 다관점 종합, 의사결정 |
| **타임아웃** | 없음 (항상 활성) |

Master의 핵심 책무:

1. **모드 감지** — 프롬프트를 분석하여 최적 모드 선택
2. **Slave 위임** — 적절한 Slave에게 작업 할당
3. **결과 판정** — Slave 결과를 비교, 검증, 병합
4. **승격/다운그레이드** — 작업 위험도에 따라 모드 자동 조정
5. **최종 실행** — 사용자 승인 후 실제 변경 수행

:::warning Master만이 변경을 실행합니다
Slave는 **분석과 제안만** 합니다. 파일 수정, 배포, 삭제 등 실제 변경은 반드시 Master가 사용자 승인을 받은 후 실행합니다.
:::

## Gemini — Speed Slave

| 항목 | 내용 |
|------|------|
| **역할** | 속도 우선 처리 |
| **강점** | 대량 데이터 분석, AWS API 호출, 로그 요약, 영향도 분석 |
| **타임아웃** | 45초 |

Gemini가 빛나는 순간:

- **10만 줄 로그**를 37초 만에 패턴 분석
- **Cost Explorer API** 12회 병렬 호출로 비용 이상 탐지
- **보안 패턴 매칭** — 넓게 빠르게 훑기
- **영향 범위 분석** — 변경이 어디까지 영향을 미치는지

### 제약 사항

- 정밀도보다 속도 우선이므로, 세밀한 로직 분석에는 Codex가 우수
- AWS API 직접 호출 시 네트워크/인증 설정 필요

---

## Codex — Precision Slave

| 항목 | 내용 |
|------|------|
| **역할** | 정밀도 우선 처리 |
| **강점** | 코드 변경, 보안 검토, 테스트 작성, 수학적 검증 |
| **타임아웃** | 90초 |

Codex가 빛나는 순간:

- **SQL injection PoC** 작성으로 보안 취약점 실증
- **state mv 스크립트** 생성으로 Terraform 안전 마이그레이션
- **유닛 테스트 156개** 일괄 생성 (커버리지 23% → 84%)
- **Error Budget 수학적 계산** — 소수점까지 정확

### 제약 사항

- Sandbox 환경에서 동작하므로 **네트워크 접근 제한**
- AWS API 직접 호출이 어려움 → 대량 조회는 Gemini에 위임
- 속도보다 정확성 우선이므로, 시간 긴급한 작업에는 부적합

---

## Kiro — Spec Slave

| 항목 | 내용 |
|------|------|
| **역할** | 설계 / 스펙 생성 |
| **강점** | 요구사항 분석, 아키텍처 설계, 태스크 분해, 체크리스트 생성 |
| **타임아웃** | 120초 |

Kiro가 빛나는 순간:

- **Terraform 모듈 분리 설계** — 변경 빈도 분석 기반 모듈 경계 결정
- **DR 런북 작성** — 실제 엔드포인트가 포함된 즉시 실행 가능 문서
- **마이크로서비스 경계 결정** — 의존성 그래프 분석 + 이벤트 설계
- **P1 장애 트리아지** — 5-Layer 체계적 점검 프레임워크

### 제약 사항

- 코드 실행/검증은 역할 외 → Codex에 위임
- 단순 설정 변경에는 오버헤드 → `@design → @ask` 다운그레이드 가능

## 에이전트 간 상호작용

### @verify에서의 역할 분담

```
Gemini (넓게, 빠르게)     Codex (깊게, 정확하게)
├ 패턴 매칭 기반 감지       ├ 로직 추적 기반 분석
├ 6개 이슈 3초 만에 발견    ├ PoC 작성 + 수정안 제시
└ 컨벤션/스타일 감지        └ 인증 누락 등 깊은 이슈 발견
         │                           │
         └─────────┬─────────────────┘
                   ▼
           Master (병합 판정)
           ├ 각 이슈별 더 적절한 쪽 채택
           ├ 두 결과 병합하여 최적 버전 생성
           └ 4-Block Format으로 최종 출력
```

### @mobilize에서의 역할 분담

```
Step 1: Kiro ──→ "어디를 봐야 하는가" (트리아지, 체크리스트)
         │
Step 2: Codex ──→ "무엇이 잘못되었는가" (근본 원인 특정)
         │
Step 3: Gemini ──→ "어디까지 영향이 있는가" (영향도 분석)
         │
Master ──→ Go/No-Go 최종 판정 (3관점 통합)
```
